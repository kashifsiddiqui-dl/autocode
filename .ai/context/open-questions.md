## 1. Exact Embedding Model Benchmarking Needed

**Context**: The current plan uses OpenAI `text-embedding-3-large` (3072 dimensions) as the primary embedding model. However, no empirical benchmarking has been done against the ICD-10-CM dataset to validate this choice.

**Question**: Which embedding model provides the best retrieval quality for ICD-10-CM code search, considering the tradeoff between quality, cost, latency, and dimensionality?

**Candidates to Benchmark**:
- OpenAI `text-embedding-3-large` (3072 dims, $0.13/1M tokens)
- OpenAI `text-embedding-3-small` (1536 dims, $0.02/1M tokens)
- `sentence-transformers/all-MiniLM-L6-v2` (384 dims, free/self-hosted)
- `sentence-transformers/all-mpnet-base-v2` (768 dims, free/self-hosted)
- Medical-domain models: `BiomedNLP/BiomedBERT`, `medicalai/ClinicalBERT`

**Evaluation Criteria**:
- Recall@10 on benchmark test set
- Latency (embedding generation time per query)
- Cost per 1M tokens
- Dimensionality impact on Qdrant storage and search speed

**Blocked By**: Need a curated benchmark test set of clinical descriptions with ground-truth ICD-10-CM codes before meaningful comparison is possible.

**Target Resolution**: Phase 2 (Data Ingestion Pipeline), after initial ingestion is working with any model.

## 2. Qdrant HNSW Tuning Parameters

**Context**: Qdrant uses HNSW (Hierarchical Navigable Small World) as its approximate nearest neighbor index. The default parameters work for general use cases, but medical coding may benefit from tuning.

**Question**: What HNSW parameters (m, ef_construct, ef) provide the best recall/latency tradeoff for our ~72,000 code collection?

**Parameters to Tune**:
- `m` (number of edges per node): Default 16. Higher values improve recall but increase memory and build time. Range to test: 16, 32, 48, 64.
- `ef_construct` (search depth during index construction): Default 100. Higher values improve index quality. Range to test: 100, 200, 400.
- `ef` (search depth during query): Default 128. Higher values improve query recall at the cost of latency. Range to test: 64, 128, 256, 512.

**Consideration**: With only ~72,000 vectors, the collection is relatively small. HNSW defaults may be sufficient, and exact search might even be feasible (< 100ms). Benchmarking should confirm whether tuning is necessary or if defaults are good enough.

**Blocked By**: Need the collection loaded with real embeddings before tuning is meaningful.

**Target Resolution**: Phase 3 (RAG Retrieval), after collection is loaded and the benchmark framework is operational.

## 3. Export PDF Template Design

**Context**: Users will export coding results as PDF documents. These PDFs need to be professional, branded, and include all relevant coding information in a readable format.

**Question**: What should the PDF export template look like? What content, layout, and branding elements should it include?

**Considerations**:
- **Header**: Auto Code logo, organization name (tenant), export date, user name.
- **Patient Context** (if provided): MRN, encounter ID, date of service. This is PHI and must be handled carefully.
- **Session Summary**: Clinical description input, number of codes found, date/time of coding session.
- **Code Table**: Ranked list of assigned codes with: code, short description, long description, confidence score, hierarchy path, includes/excludes notes.
- **Footer**: Disclaimer ("This output is generated by an AI-assisted coding system and should be reviewed by a certified medical coder"), page numbers, generated timestamp.
- **Formatting**: Portrait orientation, standard medical document typography, color coding for confidence levels.

**Library Options**:
- `reportlab` (Python) -- Full control, complex API.
- `weasyprint` (Python) -- HTML/CSS to PDF, easier templating.
- `puppeteer`/`playwright` (headless browser) -- Most flexible but heaviest.

**Blocked By**: No design mockup yet. Need input from product/design on branding and layout preferences.

**Target Resolution**: Phase 4 (API & Frontend), when the export feature is implemented.

## 4. HL7 FHIR Version to Support

**Context**: The export feature has a planned (future) HL7 FHIR export option. FHIR (Fast Healthcare Interoperability Resources) is the standard for healthcare data exchange, and exporting coding results as FHIR resources would enable integration with EHR systems.

**Question**: Which FHIR version should we target, and which FHIR resources should the export produce?

**FHIR Version Options**:
- **FHIR R4 (4.0.1)**: The current most widely adopted version. Most EHR systems support R4. Safe default choice.
- **FHIR R4B (4.3.0)**: Minor update to R4 with some additions. Backward-compatible with R4.
- **FHIR R5 (5.0.0)**: Latest version with significant changes. Adoption is still early. Riskier to target.

**FHIR Resources to Generate**:
- `Condition` -- Each assigned ICD-10-CM code as a Condition resource with `code.coding.system = "http://hl7.org/fhir/sid/icd-10-cm"`.
- `DiagnosticReport` -- Wrapper resource containing the coding session results.
- `Encounter` -- If encounter context is provided.
- `Bundle` -- Transaction bundle containing all resources.

**Additional Questions**:
- Should we support FHIR JSON, XML, or both? (JSON is more common in modern implementations.)
- Do we need to validate against US Core profiles?
- Should the export be downloadable or pushed to an endpoint (FHIR server)?

**Blocked By**: Need clarity on target customer EHR integration requirements. This is a Phase 5+ feature.

**Target Resolution**: Phase 5 or later, based on customer feedback and integration requirements.

## 5. Rate Limiting Thresholds

**Context**: Rate limiting is needed at multiple levels (nginx, WAF, application) to prevent abuse, ensure fair usage, and protect against DoS attacks. The thresholds need to balance security with usability.

**Question**: What rate limiting thresholds should be configured at each level?

**Levels and Proposed Defaults** (need validation):

| Level | Scope | Proposed Limit | Notes |
|-------|-------|---------------|-------|
| nginx | Per IP, all endpoints | 100 req/min | General abuse prevention |
| WAF | Per IP, all endpoints | 1000 req/5min | DDoS protection |
| Application | Per user, coding endpoint | 30 req/min | LLM cost protection |
| Application | Per user, export endpoint | 10 req/min | Resource-intensive |
| Application | Per tenant, coding endpoint | 200 req/min | Tenant-level fair usage |
| Application | Per tenant, all endpoints | 1000 req/min | Tenant-level overall |

**Considerations**:
- The coding endpoint is the most expensive (triggers LLM calls). Rate limiting here directly controls cost.
- Power users in a busy coding session might submit 10-20 queries in rapid succession. The per-user limit must accommodate this.
- Different subscription tiers may have different limits (future).
- Rate limit responses should include `Retry-After` header and return HTTP 429.
- Need to decide: hard reject vs. queue/throttle when limits are hit.

**Blocked By**: No usage data yet (pre-launch). Initial thresholds should be conservative and adjustable without redeployment (config-driven).

**Target Resolution**: Phase 4 (API & Frontend), refined after initial user testing.
